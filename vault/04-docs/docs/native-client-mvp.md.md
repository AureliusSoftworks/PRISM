# Prism Native Client MVP

This document scopes the first official Prism iOS/Mac client. The goal is a
small native vertical slice that proves discovery, pairing, authenticated API
calls, conversation loading, and one chat send against a real Prism Server.

## Platform Targets

- iOS first.
- Mac Catalyst or native macOS second, using the same API client and session
  model where practical.
- No backend services run on iOS or iPadOS.

## App States

The native client should be modeled around a small set of explicit states:

| State | Meaning | Primary UI |
| --- | --- | --- |
| `needsServer` | No server selected and no stored session | Welcome and setup explainer |
| `discovering` | Local Network permission requested and scan active | Server discovery list |
| `manualEntry` | User is entering a host manually | URL form and validation |
| `needsPairing` | Server selected but device not trusted yet | QR scanner or pairing code |
| `authenticated` | Valid Keychain session exists | Conversation list and chat |
| `locked` | App is protected locally | Face ID/Touch ID unlock |
| `offline` | Stored server cannot be reached | Reconnect, edit server, retry |

These states should be explicit in the app model instead of inferred from many
optional values. That keeps the onboarding understandable and prevents fragile
edge cases around partially paired devices.

## First-Run Flow

1. Show a short welcome screen: "Prism connects to your Prism Server."
2. Ask for Local Network access before scanning.
3. Discover servers advertised as `_prism._tcp.`.
4. Show each server as a tappable row with name, host, readiness, and pairing
   status.
5. Offer manual URL entry below the discovery list.
6. After server selection, validate `GET /api/health`.
7. Pair with a QR code or short code generated by Prism Server.
8. Store the bearer session token in Keychain.
9. Load `GET /api/auth/me`, then `GET /api/conversations`.
10. Open the conversation list or new-chat screen.
