# Prism Native Client MVP

This document scopes the first official Prism iOS/Mac client. The goal is a
small native vertical slice that proves discovery, pairing, authenticated API
calls, conversation loading, and one chat send against a real Prism Server.

## Platform Targets

- iOS first.
- Mac Catalyst or native macOS second, using the same API client and session
  model where practical.
- No backend services run on iOS or iPadOS.

## App States

The native client should be modeled around a small set of explicit states:

| State | Meaning | Primary UI |
| --- | --- | --- |
| `needsServer` | No server selected and no stored session | Welcome and setup explainer |
| `discovering` | Local Network permission requested and scan active | Server discovery list |
| `manualEntry` | User is entering a host manually | URL form and validation |
| `needsPairing` | Server selected but device not trusted yet | QR scanner or pairing code |
| `authenticated` | Valid Keychain session exists | Conversation list and chat |
| `locked` | App is protected locally | Face ID/Touch ID unlock |
| `offline` | Stored server cannot be reached | Reconnect, edit server, retry |

These states should be explicit in the app model instead of inferred from many
optional values. That keeps the onboarding understandable and prevents fragile
edge cases around partially paired devices.

## First-Run Flow

1. Show a short welcome screen: "Prism connects to your Prism Server."
2. Ask for Local Network access before scanning.
3. Discover servers advertised as `_prism._tcp.`.
4. Show each server as a tappable row with name, host, readiness, and pairing
   status.
5. Offer manual URL entry below the discovery list.
6. After server selection, validate `GET /api/health`.
7. Pair with a QR code or short code generated by Prism Server.
8. Store the bearer session token in Keychain.
9. Load `GET /api/auth/me`, then `GET /api/conversations`.
10. Open the conversation list or new-chat screen.

## Screen MVP

### Welcome

Purpose: explain the required Prism Server without making the user feel blocked.

Required actions:

- Continue to server discovery.
- Open setup help for downloading Prism Server from GitHub Releases.

### Server Discovery

Purpose: find a local Prism Server.

Required behavior:

- Request Local Network permission.
- Scan for Bonjour/mDNS `_prism._tcp.` services.
- Show discovered servers as native rows.
- Allow refresh.
- Allow manual host entry.

### Pairing

Purpose: trust the device with a server-side session.

Required behavior:

- Show selected server identity before pairing.
- Accept QR scan or typed pairing code.
- Exchange the pairing nonce for a session token.
- Store the session token in Keychain.
- Handle expired or reused pairing codes with a clear retry path.

### Conversations

Purpose: prove authenticated read access.

Required behavior:

- Fetch `GET /api/conversations`.
- Display title, updated date, bot accent when available, and empty state.
- Open a conversation by fetching `GET /api/conversations/:id`.

### Chat

Purpose: prove authenticated write access.

Required behavior:

- Send `POST /api/chat` with the selected conversation or a new conversation.
- Render the returned assistant message and updated conversation.
- Show provider/model metadata when supplied.
- Surface server errors without losing the typed message.

### Settings Lite

Purpose: expose only what the MVP needs.

Required behavior:

- Show connected server name and URL.
- Show signed-in Prism account from `GET /api/auth/me`.
- Disconnect this device by invalidating the session.
- Enable Face ID/Touch ID app lock.

## Native Service Boundaries

Keep the first client architecture small:

- `PrismApiClient`: typed HTTP client for `/api/*`.
- `ServerDiscoveryService`: Bonjour/mDNS scan and manual URL validation.
- `PairingService`: QR/code exchange and pairing errors.
- `SessionStore`: Keychain-backed token and selected server metadata.
- `AppLockService`: Face ID/Touch ID gate.

Avoid putting networking directly in SwiftUI views. Views should react to app
state and call small services.

## Minimum Data Models

Start with Swift equivalents of the shared TypeScript models:

- `UserProfile`
- `Conversation`
- `ChatMessage`
- `ChatRequestPayload`
- `ChatResponsePayload`
- `ChatMode`
- `BotSummary` or a minimal bot DTO from `GET /api/bots`

The native app should not define its own conversation semantics. Server
responses are the contract.

## Vertical Slice Completion Criteria

The MVP is complete when a fresh install can:

1. Discover or manually connect to a local Prism Server.
2. Pair and store a Keychain session.
3. Fetch the signed-in user and conversation list.
4. Open a conversation.
5. Send one chat message and render the assistant response.
6. Disconnect the device and prove the old token no longer works.

Do not expand into the full Sandbox/customizer surface until this slice works
on a real device over a real local network.
